Case Study · 03

Customer Feedback Classification

Building a robust multiclass NLP classifier to route 10,000 customer comments across 28 product departments — under severe class imbalance and real-world distribution shift.

Context

Academic · Group Project (COMP9417)

Role

EDA, Feature Selection & Model Stacking

Stack

Python · Scikit-learn · XGBoost · LightGBM

Dataset

10,000 NLP-encoded comments · 300 features · 28 classes

01 — Business Problem

Routing customer feedback at scale

A manufacturing company receives thousands of customer comments daily across 28 product lines. Manually triaging each comment to the right department is slow and error-prone. The task: train a model to automatically classify each comment and route it to the correct team, minimising misrouted feedback and wasted response time.

Each comment had been encoded into a 300-dimensional NLP feature vector — but with no feature names or semantic labels attached, making domain interpretation impossible. Two additional challenges made this harder than a standard classification task: severe class imbalance (one class holding 45% of all samples) and distribution shift between the training set and a second deployment test set.

Evaluation metric: Weighted cross-entropy loss — where rare classes receive higher weight via inverse frequency. This explicitly penalises ignoring minority classes, making standard accuracy a misleading proxy throughout.

02 — Exploratory Data Analysis

Understanding the data before touching a model

With 300 anonymous NLP features and no domain labels, EDA had to do the heavy lifting. Three distinct analyses shaped the modelling strategy.

Class imbalance — the defining challenge

The class distribution is dramatically skewed. Class 5 alone accounts for nearly half the training data, while several classes have fewer than 10 samples. This makes accuracy a near-useless metric — a model predicting class 5 every time would still achieve ~45% accuracy. Weighted log loss was used instead to force genuine minority class performance.

Feature quality — mostly clean, but skewed

Feature means clustered near zero — confirming pre-standardized NLP embeddings. Standard deviations were uniformly narrow, suggesting many features may be low-signal. Outlier detection via Z-score (|z| > 3) flagged fewer than 0.4% of rows across all features — too sparse to warrant removal. The top 20 most skewed features showed right-skew values between 1.2 and 4.5, consistent with rare NLP pattern activations. Tree-based models were chosen partly because they handle skewed features without transformation.

Pairwise Pearson correlation analysis of all 300 features found no pair exceeding r = 0.69 — well below the 0.95 threshold typically used for redundancy removal. All features were retained for feature selection to decide.

Distribution shift — Test Set 2 is different

Comparing feature-level statistics across the training set, Test Set 1, and Test Set 2 revealed a clear pattern: feature means, standard deviations, and value ranges diverged substantially in Test Set 2 relative to training. This is a textbook covariate shift — the input distribution P(x) has changed, while the underlying class-conditional P(y|x) may still hold.

03 — Feature Selection

From 300 features to 105 that matter

With 300 anonymous features and no semantic guide, feature selection required empirical ranking. Three methods were run in parallel — permutation importance with Random Forest, permutation importance with Logistic Regression, and XGBoost native feature importance — each producing a top-50 ranking. The union of all three sets gave 105 features for the final models.

05 — Handling Class Imbalance

Why accuracy is the wrong metric

The dataset's dominant Class 5 (4,479 of 10,000 samples) creates a deceptive accuracy landscape — a naive majority-class predictor achieves ~45% accuracy. Three strategies were applied to address this throughout the pipeline.

01

Weighted loss metric

The evaluation metric itself applies inverse-frequency class weights — rare classes receive proportionally higher penalty when misclassified. This forced all model choices to account for minority class performance, not just dominant class accuracy.

02

class_weight='balanced'

Applied uniformly to all sklearn models. This adjusts sample weights during training so each class contributes equally to the loss update — preventing the model from collapsing to always predict Class 5.

03

Stratified splitting

All train/test splits used stratified sampling. With classes as rare as 7 samples, non-stratified splits risked excluding entire classes from training or evaluation sets — making stratification critical rather than optional.

04

Per-class F1 monitoring

Final model evaluation used per-class precision, recall, and F1 to surface minority class failures that aggregate scores hide. Classes with F1 = 0.00 (e.g., Class 0, 1, 2) revealed where even the best model struggled.

06 — Distribution Shift (Test Set 2)

Deploying to a different distribution

Test Set 2 presented a simulated real-world deployment scenario: 202 labeled points and 1,818 unlabeled points, with feature statistics that diverged measurably from training. This is covariate shift — the feature distribution P(x) changes, but the class-conditional relationship P(y|x) may still hold.

What the shift looks like

Feature-level mean, standard deviation, and min/max ranges all showed visible divergence in Test Set 2 versus the training set. Test Set 1 tracked training closely, while Test Set 2 showed systematic differences across hundreds of features simultaneously.

Mitigation strategies explored

The 202 labeled Test Set 2 samples were used to validate adapted models. Ensemble regularisation and lower-complexity models (Logistic Regression) showed better cross-distribution stability than high-capacity tree models that overfit training-specific patterns.

Key finding: The stacked model's generalization advantage was amplified under distribution shift — the meta-learner's smoothing effect reduced sensitivity to individual base model failure modes specific to the training distribution.

The LightGBM validation accuracy of 76.95% and weighted F1 of 0.75 show the model performed well on majority classes. The stacked architecture's true advantage appeared in the train-test WLL gap: a difference of just 0.0006 compared to Random Forest's gap of 0.014 — a 20× improvement in generalization consistency.

Per-class inspection revealed persistent failures on ultra-rare classes (0, 1, 2, 22) with fewer than 10 samples — a structural limitation no model selection strategy can fully overcome without additional data.

08 — Limitations & Reflections

What we'd do differently

  • Anonymous features limit interpretability. With 300 unlabelled NLP dimensions, it was impossible to construct domain-informed features or understand why specific features ranked highly. Raw NLP embeddings carry semantic structure that permutation importance can rank but not explain.
  • SMOTE explored but not used. Synthetic oversampling of rare classes was considered to address minority class imbalance. However, with classes as small as 7 samples, synthetic augmentation risked amplifying noise rather than signal, and stratified class weighting was used instead. A more targeted SMOTE strategy on a larger dataset could materially improve minority class recall.
  • Distribution shift mitigation was limited. While covariate shift was diagnosed and ensemble regularisation provided partial defence, dedicated domain adaptation methods (CORAL, importance reweighting, fine-tuning on the 202 labeled Test Set 2 points) were not fully explored.
  • Hyperparameter tuning scope. GridSearchCV covered 72 LightGBM parameter combinations but the search space for the stacking meta-learner and XGBoost was more limited. Bayesian optimization could have surfaced better configurations more efficiently.
